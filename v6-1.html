The Engineering Question

How Do Swarms Actually Work?

Not "what are agents" — your engineers already know that. The question is: what happens when you need 5 agents working simultaneously on different parts of a problem, each with its own context window, each needing access to institutional knowledge, and all of them needing to produce a coherent result?

The Naive Approach
1

One big agent does everything

Fill one context window with everything. It works for 20 minutes. Then context degrades. At 128K tokens, GPT-4 shows 15.4% performance loss. Instructions at the beginning vanish. The agent starts hallucinating its own earlier output.

The Common Approach
N

N identical agents vote on answers

Run 3 copies of GPT-4, take majority vote. Reaches 82% accuracy. Sounds reasonable — until you learn that 3 diverse medium-capacity models reach 91%, beating GPT-4. Identical agents form echo chambers.

What We Built
Mesh

Specialized agents, adversarial review, shared memory

Conductor stays thin. Workers get clean context windows. KB knowledge injects into every sub-agent. Adversarial observers catch what workers miss. Cross-session mesh enables live coordination. 100% correction rate.

Research Foundation
UC Berkeley RouteLLM + Anthropic Multi-Agent System + arXiv 2410.12853
Three independent results converge: 3.66x cost reduction via model routing (Berkeley), 90.2% performance improvement from parallel sub-agents over single-agent (Anthropic), and 91% accuracy from diverse models vs. 82% from identical models (Diversity of Thought). The architecture is not a design preference — it's what the math demands.

Layer 1 — Auto-Swarm Decomposition

How Tasks Become Parallel Agents.

Every non-trivial task goes through the same pipeline: classify, decompose, enrich, launch, synthesize. The classifier determines the pattern. The enricher injects institutional knowledge. The launcher manages parallelism. The synthesizer combines results.

1

Classify

The task hits the classifier. First, a learned router (embeddings from prior sessions) tries to match. If confidence < 0.40, falls back to keyword heuristics. Output: research build or analysis.

2

Plan Agents

Pattern selects an agent topology. Research: KB-researcher + Web-researcher + Codebase-scanner. Build: Researcher + Architect. Analysis: Investigator + Pessimist + Pragmatist. Each gets a specific role and model tier.

3

KB Injection

Every sub-agent prompt is prepended with m4_knowledge_header() — 300-500 tokens containing: M4 identity, top 3 domain-relevant mistakes from KB, top 3 patterns. This is how institutional memory travels with every agent.

4

Launch

All independent agents launch simultaneously in background. Each writes detailed output to a designated file path. Returns only a 2-5 sentence summary to the conductor. The conductor stays thin — sub-agents get full context windows.

5

Synthesize

When all parallel agents complete, a synthesizer reads all output files and produces a unified result: executive summary, recommended approach, key constraints, risks, action items. This runs foreground so the conductor gets the result.

The cost calculation: Three haiku sub-agents running in parallel cost ~$0.003 total. One sonnet agent doing the same work costs ~$0.04. One opus agent costs ~$0.15. The research pattern uses 3 haiku agents for a 50x cost reduction vs. opus — while producing better results because each agent gets a clean, focused context window instead of one bloated one.

Layer 2 — The Forge Pipeline

Worker + Observer + Gate.

The Forge is not "code review." It's an adversarial quality gate where the Worker and Observer share zero context. The Worker builds. The Observer attacks. The Gate validates. They communicate through a file-based protocol designed for crash resilience.

Forge Communication Protocol
WORKER writes to progress.log:
  [STEP 1] Parse input requirements
  <implementation content>
  ---END STEP---

OBSERVER polls progress.log every 5 seconds.
  On new step → evaluate against domain checklist + perspective:
    Perspectives: security | performance | maintainability

  If issues found:
    → append to corrections.inbox:  [CORRECTION] Use parameterized queries, not string concat
    → append to critique.log:       Detailed reasoning
    → append to pending-kb-saves.jsonl: Save finding for future sessions

  Always → append to approvals.log:  [APPROVED] Step 1

WORKER reads corrections.inbox before each new step.
  → Applies all corrections
  → Clears inbox (truncate)
  → Proceeds to next step

SYNTHESIS OBSERVER waits for all perspectives to approve:
  → Writes [APPROVED-FINAL] Step N
  → Worker advances only on FINAL approval
Worker

Builds the artifact step-by-step

Reads corrections before each step. Writes cumulative output to output/ after each step (protects against crash). Polls approvals.log for approval. 90-second timeout — advances with warning if Observer stalls.

Observer(s)

Attacks with fresh context

Up to 3 perspectives (security, performance, maintainability). Each has its own critique log and corrections inbox. Zero shared context with Worker — different model, different priors, different failure modes.

Gate

Validates the corrections

Synthesis observer reads all per-perspective approvals. Only writes [APPROVED-FINAL] when all perspectives approve. The Worker cannot advance past the gate without consensus from adversarial reviewers.

Why 100% correction rate: The key insight from arXiv 2410.12853 — three identical agents reach 82% accuracy. Three diverse agents reach 91%. The Forge goes further: the Observer is explicitly adversarial. It doesn't validate — it attacks. Different model, fresh context window, no attachment to the Worker's output. This is adversarial separation, not consensus verification. 100% correction rate across Bash, Python, Docker Compose, and REST API domains.

Layer 3 — The Neural Mesh

Cross-Session Awareness.

The mesh is how multiple agent sessions — each with its own context window, running in parallel terminals — become aware of each other. It's the nervous system that prevents two agents from editing the same file, enables one agent's discovery to benefit all others, and coordinates handoffs when sessions approach context limits.

Three Communication Primitives

Broadcast Channel

Append-only JSONL

One-to-many. Every session publishes discoveries, decisions, questions, and updates to feed.jsonl. Other sessions check via mtime comparison (~1ms) every 10 tool calls. Injected as <cross-session-broadcast> into Claude's context. 1-hour TTL, compacted at 200 entries. Also ingested to KB for permanent memory.

Mesh Digest

Atomic JSON snapshots

Per-session state written every 15 seconds via atomic .tmp + rename(). Contains: files_touched, decisions, discoveries, errors, open_questions, progress (0-1). The ConflictDetector reads all digests and emits file_conflict, duplicate_work, and discovery signals.

Signal Channels

Three types of file-based messaging

LogChannel: Append-only structured log. MailboxChannel: Read-and-clear messaging (corrections, commands). SignalChannel: Permanent coordination signals ([SIGNAL] agent DONE). All file-backed, crash-resilient, replayable.

Broadcast Entry (actual structure)
{
  "ts": "2026-02-24T12:00:00Z",
  "session_id": "20260224-091500-12345",
  "category": "discovery",     // discovery | decision | question | request | update
  "title": "JWT RS256 required for separate deployments",
  "content": "Auth and API are separate trust boundaries — symmetric HS256 can't work",
  "kb_id": 42              // Also saved to persistent KB

// Mesh Digest (per session, every 15 seconds)
{
  "session_id": "20260224-091500-12345",

How It Actually Fires

Every Tool Call
Post-tool hook fires (~15ms)
Increments call counter. Estimates byte consumption. Checks human signal file. Every 10 calls: checks broadcast feed via mtime comparison (~1ms when unchanged).
Every 15 Seconds
Mesh sentry writes digest
Background daemon reads session's event stream. Writes atomic JSON snapshot to scratchpad/mesh/<session_id>.json. ConflictDetector compares all digests, emits signals for file conflicts or duplicate work.
On Git Push
Auto-broadcast to all sessions
Post-tool hook detects git push. Fires background subprocess: extract commit hash, message, file list. Publishes to broadcast feed. Other sessions get <code-pushed> injection with file overlap detection.
On Dependency Failure
Cross-session recovery
When a tool fails, the hook records it. When a peer session broadcasts with restore keywords ("enabled", "fixed", "available"), the hook injects <dependency-restored> instead of a generic broadcast. The failing session retries automatically.

Engineering Detail — The File-as-Message-Bus Architecture

Why Files, Not Queues?

Redis/RabbitMQ

Another dependency to fail

Adds a process to keep alive. Connection pooling. Serialization overhead. If the broker dies, all inter-agent communication dies. Agents can't inspect the message history. No replay. No crash recovery without persistent storage.

HTTP/gRPC

Tight coupling, discovery problem

Requires service discovery. Agents need to know each other's addresses. Synchronous calls block. Connection failures cascade. Health checks add complexity. Timeouts need tuning. No offline inspection.

Filesystem

Zero dependencies, total observability

Always available. Atomic writes via .tmp + rename(). Every message is inspectable (cat feed.jsonl). Full replay. Crash recovery is free — partial writes just get overwritten. tail -f for real-time monitoring. No serialization overhead beyond JSON.

Three Channel Primitives
LogChannel — Append-only structured log
  Write:  append "[ENTRY 42] Step complete\n<body>\n---END ENTRY---"
  Read:   parse all entries by marker pattern
  Use:    progress.log, activity.log, critique.log

MailboxChannel — Read-and-clear one-shot messaging
  Write:  append line to file
  Read:   read all lines, truncate file to empty
  Peek:   read without clearing
  Use:    corrections.inbox (Worker reads, clears, applies)

SignalChannel — Permanent coordination signals (never cleared)
  Write:  append "[SIGNAL] kb-researcher DONE"
  Check:  substring match on any signal
  List:   return all signals
  Use:    signals.log (track agent completion, coordination gates)
Context Pressure — Dual-Signal Enforcement
// Per-tool byte estimation (from post-tool-use hook)
Read     → actual file size (capped 100KB)
Task     → 80KB (sub-agent returns)
Grep     → 12KB
Bash     → 10KB
WebFetch → 15KB
KB ops  → 8KB
Other   → 1KB

// Pressure tiers (tool call count + byte estimate)
NORMAL   (0-50%)   All hooks fire at base frequency
ELEVATED (50-70%)  Broadcast/mesh frequency reduced 2x
HIGH     (70-85%)  Suppress broadcasts, mesh, non-essential injections
CRITICAL (85%+)   Only relay signal + human signals + circuit breaker

// Enforcement gates (one-shot, milestone-based)
Call 10+   → KB search gate: "Have you searched KB yet?"
Call 20+   → Early warning: if >70% heavy tools, nudge delegation
Call 30    → Delegate: new heavy work goes to sub-agents
Call 60    → Mandatory conductor mode
Call 90    → Strict wind-down: summarize, prepare handoff
Call 150+  → FORCE RELAY SPAWN
The Evolution — From Prototype to Production Neural Mesh

Where This Goes.

Today's mesh runs on a single machine with filesystem communication. The architecture is designed to evolve. Each layer was built to be replaceable without changing the layers above or below it. Here's the trajectory.

Now — Single Machine Mesh
Filesystem-as-message-bus. Proven at 3-5 concurrent sessions.
JSONL broadcasts, atomic JSON digests, file-based channels. 15ms hook budget. 15-second mesh refresh. All communication through scratchpad/. Works because Apple Silicon M4 Max handles all concurrent sessions on one machine with 64GB unified memory. Zero network latency. Total observability. The file paths in this presentation are real file paths on a real machine.
Phase 2 — StarRocks as Mesh Backbone
Replace filesystem with real-time database. Same protocols, distributed scale.
Broadcasts → StarRocks table with TTL-based compaction. Mesh digests → materialized views with 15-second refresh. Signal channels → StarRocks queue tables with exactly-once semantics. The channel abstractions (LogChannel, MailboxChannel, SignalChannel) get new storage backends. The Forge, auto-swarm, and orchestration patterns don't change at all — they just read and write through the same interfaces.
Phase 3 — Multi-Machine Mesh
Agents on different machines, different clouds, same mesh awareness.
StarRocks federates the mesh across regions. Broadcast entries replicate with sub-100ms latency (proven at 10,000+ QPS). Mesh digests become globally visible. ConflictDetector runs as a StarRocks continuous query. The post-tool hook swaps filesystem mtime checks for HTTP HEAD against StarRocks API — same ~1ms budget. Agent sessions can run on different machines, different cloud instances, even different model providers.
Phase 4 — Customer-Facing Agent Mesh
Enterprise customers run their own agent meshes. CelerData is the backbone.
Each customer gets an isolated mesh namespace in StarRocks. Their agents publish discoveries, coordinate work, learn from mistakes — exactly like the prototype, but at enterprise scale with row-level security, audit trails, and compliance gates. The learning loop means each customer's mesh gets smarter over time. The moat compounds: switching costs increase with institutional knowledge stored in the mesh. This is the product — not the agents, not the models. The mesh.
