Answering James's Question — Three Designs or One Ladder?

Three Memory Tiers.
They're Not the Same.

James identified three designs and asked if design 3 is the same as design 2. The answer: no. They are an ascending ladder. Each tier includes the one below it, then adds the feedback loops that make the difference between a database and an intelligence.

Tier 1

Project Memory — What Claude Code Does

Auto-Memory records patterns from conversations. Scoped to one project, one user. No scoring, no learning, no cross-pollination. Like a personal notebook. CLAUDE.md and MEMORY.md files. Useful but limited — no way to know which memories are valuable, no way to share across agents.

↓
Tier 2

Shared Knowledge Base — What CelerData Is Building

PostgreSQL + pgvector (or S3 + Kafka + StarRocks). Shared across users and agents. Data flows IN from pipelines. Agents read OUT via search. One direction. Like a company wiki. James's text2sql example lives here: record question + SQL + tables used. Any agent can retrieve it. But the wiki doesn't know which entries are good.

↓
Tier 3

The Learning Loop — What We Built on Top

Session outcomes write back automatically. MemRL utility scoring — knowledge ranked by usefulness. ExpeL rule extraction — patterns mined from experience. Mistake tracking — same error never happens twice. Neural router — trained on session outcomes, improves routing. Like a colleague who remembers everything, learns from mistakes, and gets better every day.

The ladder is the product. Tier 1 is free (Claude gives it away). Tier 2 is infrastructure (StarRocks, pgvector — table stakes). Tier 3 is what compounds. A competitor can build Tier 2 in weeks. They cannot replicate thousands of sessions of utility-scored, domain-specific, mistake-aware knowledge. That's the moat.

The Prompt Intelligence Engine

4 Threads. <2 Seconds.

Every time a user hits Enter, a hook fires before the model sees the prompt. Four parallel threads run simultaneously: KB preflight, skill detection, the learning engine, and deep cognitive reasoning. Their outputs merge into a single intelligence injection that transforms a naive prompt into an informed one.

1

KB Preflight

Hybrid search: pgvector cosine + full-text, fused via RRF. Returns domain mistakes, known patterns, repeat error flags. GPU rerank via qwen3:8b. ~800ms.

2

Skill Detection

LearnedRouter (MLX neural net) trained on session outcomes. Falls back to keyword matching if confidence <0.40. Returns ranked skill matches. ~200ms.

3

Learning Engine

Three sub-engines: ReflexionEngine (domain reflections), ExperienceEngine (high-confidence rules), CoreMemoryEngine (self-edit check). ~300ms combined.

4

Deep Reasoning

Single qwen3:8b call: intent type, reasoning mode, emotion detection, curiosity questions. Local GPU, $0.00. ~400ms.

The Seven Learning Mechanisms

The Compound Engine.

Each mechanism addresses a different timescale of learning. Together, they create a system where every interaction improves future interactions — not linearly, but compoundingly.

1
MemRL — DeepMind

Utility-Scored Memory

Every KB entry has a utility_score updated via EMA. Helpful retrieval → score up. Unhelpful → decay. Temporal decay prevents stale dominance. High-utility surfaces first.

2
ExpeL — NeurIPS 2023

Rule Extraction

After task completion, reusable rules extracted: "Separate deployments = asymmetric keys". Confidence 0-1. Success → ×1.05. Failure → ×0.9.

3
Reflexion — NeurIPS 2023

Domain Self-Reflection

Reflections stored indexed by domain. New prompt in known domain → 3 most recent reflections auto-inject. Guarantees contextual awareness.

4
Code Review Learning

Bidirectional KB

New findings → KB. Recurring findings (3+) → HIGH priority. Resolved → fix pattern documented. Known issues → KNOWN REPEAT flag.

5
Neural Router — MLX

Learned Skill Routing

MLP: 1024→256→64→12. Prompt embedding → domain classification. Trained on routing_decisions table (161 labeled examples, 8 domains). AES outcome feedback closes the loop. Live — trained and routing.

6
Dynamic Checklists

Domain Enrichment

Preflight detects domain, enriches with domain-specific items from KB. Includes reasoning traps. 17 repeat error themes tracked.

7
CoreMemory

Safe Self-Editing

The system can edit its own operating instructions. Tightly constrained: append-only, never deletes, high-confidence required. Permanent behavioral change.

utility_score = EMA(score, feedback, α=0.15) × temporal_decay(age, rate=0.005, floor=0.6)
MemRL Utility Scoring — Exponential Moving Average with Temporal Decay Floor

Why 7 mechanisms, not 1: Each covers a different timescale. MemRL: milliseconds (individual retrievals). ExpeL: minutes (task completion). Reflexion: hours (domain experience). Neural Router: days (session outcomes). Code review: weeks (codebase evolution). Checklists: months (error accumulation). CoreMemory: permanent. Together, they cover every timescale of learning.

Built & Running — Not Theoretical

Unified Telemetry.
Decision → Call → Outcome → Retrain.

The learning loop isn't a diagram. It's running code. Every model call — across 14 models, 3 providers, 5 backends — logs to a single telemetry stream. Every routing decision writes to PostgreSQL. Outcomes feed back into the neural router's training data. The system gets smarter with every call.

The Pipeline — Four Stages

1

Route

record_routing_decision() writes to JSONL + PostgreSQL. Embeds the prompt via mxbai-embed-large. Returns a decision_id for linking.

2

Call

record_call(decision_id=N) logs model, provider, task type, latency, tokens, success. Links back to the routing decision.

3

Outcome

record_outcome(N, success, latency) updates the DB row. Ties model performance to the routing decision that chose it.

4

Retrain

maybe_retrain() on session end. When labeled data ≥ 50, triggers LearnedRouter.train(). MLX classifier updates in seconds.

Multi-Agent Orchestration

Auto-Swarm + The Forge.

Two complementary systems. Auto-Swarm decomposes tasks into parallel sub-agents enriched with KB context. The Forge runs adversarial quality gates. Neither is tied to a specific model vendor.

Auto-Swarm: Classify → Plan → KB Inject → Launch → Synthesize

1

Classify

Task → pattern detection: research, build, or analysis. Determines agent topology.

2

Plan

Generate agent manifest: roles, models (haiku/sonnet), prompts, output paths.

3

KB Inject

Each sub-agent prompt enriched with relevant KB context via knowledge_inject.py.

4

Launch

All independent agents spawn in parallel. Background execution, file-based output.

5

Synthesize

Synthesis agent combines all outputs. Divergent findings flagged. Final report generated.

The Forge: Worker + Observer + Gate

Worker

Builds the artifact

Executes the task step-by-step. Writes output to shared directory. Has zero visibility into what the Observer is checking. Different model, different context.

Observer

Validates each step

Reviews Worker output against the spec. Flags errors, omissions, anti-patterns. Zero shared context with Worker. Different priors, different failure modes.

Gate

Pass or fail

Interpreter reads both outputs. If Observer found issues: Worker must fix. Cycle repeats until Gate passes. 100% correction rate across Bash, Python, Docker, REST.

The Forge is NOT Claude agent teams. Claude agent teams = model-vendor parallel execution. The Forge = adversarial quality gate built ON TOP of any agent framework. The Worker and Observer have zero shared context — different models, different priors, different failure modes. This works with Claude today. It would work with GPT tomorrow. It would work with any model.

The Neural Mesh

Cross-Session Awareness.

Multiple agents run in parallel on the same codebase. The mesh gives them awareness of each other — without shared memory, without message queues, without coordination overhead. Files, not queues.

Broadcast Channel

feed.jsonl

Append-only file. Each session writes events: git push, dependency change, error resolution. Post-tool-use hook checks mtime every 10 tool calls (~1ms). New broadcasts injected as <cross-session-broadcast>.

Mesh Digest

Session Start Injection

When a new session starts, the hook injects a digest of all active sessions: who's working on what, which files they own, what they've committed. Prevents conflicts before they happen.

Signal Channels

human-input.txt

Rick can signal any session from any terminal: signal "stop and review". Hook reads and clears the file, injecting as <human-signal>. Checked after every single tool call.

How It Actually Fires

Mesh Timeline
T+0ms    Session A: git push (auth refactor, 3 files)
T+1ms    post-tool-use.sh appends to feed.jsonl: {type: "code-pushed", files: [...]}

T+800ms  Session B: tool call #40 fires post-tool-use hook
T+801ms  Hook checks feed.jsonl mtime — changed since last check
T+802ms  Hook reads new entries, detects file overlap with Session B's work
T+803ms  Injects: <code-pushed session="A"> auth refactor touched auth/jwt.py
T+804ms  Session B sees the signal, pulls latest, adjusts its approach

Why Files, Not Queues?

Message Queues (Redis, RabbitMQ)

External dependency. Crash recovery complexity. Consumer groups, ack semantics, dead letter queues. For a system running on one machine with 2-5 concurrent sessions? Massive overkill.

Append-Only Files

Zero dependencies. Atomic append via OS. mtime check: 1ms. Crash recovery: the file is the log. Works on any OS, any filesystem. UNIX philosophy: do the simplest thing that works.

Knowledge Architecture

PostgreSQL + pgvector + WAL.

Not a document store. A scored, searchable, self-improving memory system with zero-loss writes, hybrid retrieval, and GPU-accelerated reranking.

Knowledge Table Schema
CREATE TABLE knowledge (
    id            SERIAL PRIMARY KEY,
    title         TEXT NOT NULL,
    content       TEXT NOT NULL,
    category      TEXT,          -- identity | operational | patterns | mistakes
    scope         TEXT,          -- system | family | personal | child
    tags          TEXT[],
    embedding     vector(1024), -- mxbai-embed-large (local, free)
    utility_score FLOAT DEFAULT 1.0,     -- MemRL EMA
    retrieval_count INT DEFAULT 0,
    owner_id      UUID REFERENCES users(id)  -- RLS enforcement
);

CREATE INDEX ON knowledge USING ivfflat (embedding vector_cosine_ops);
CREATE INDEX ON knowledge USING gin (to_tsvector('english', title || content));
CREATE INDEX ON knowledge (category, utility_score DESC);

Hybrid Search: Semantic + Full-Text + RRF + GPU Rerank

Semantic

pgvector cosine

Query → embedding via mxbai-embed-large. Captures meaning: "authentication" matches "login system." Top-20 candidates.

Full-Text

tsvector rank

ts_rank_cd() on title+content. Captures exact terms: "RS256" matches "RS256." Critical for code and error messages. Top-20.

RRF + GPU Rerank

Reciprocal Rank Fusion

RRF(d) = ∑ 1/(k + rank_i(d)), k=60. Merges lists without score normalization. Top-10 → qwen3:8b reranker. Final top-5.

Write-Ahead Log — Zero Knowledge Loss

KB WAL Protocol (lib/kb_wal.py)
1. WRITE: Append to pending.jsonl (content-hash dedup via SHA256)
2. REPLAY: Process pending.jsonl → PostgreSQL (upsert + embedding)
3. RECOVER: On startup, replay un-replayed entries

Handles: crash during write, DB down, embedding failure
Guarantee: if it hit pending.jsonl, it WILL reach PostgreSQL
Same pattern as PostgreSQL WAL, SQLite WAL, Kafka append-only log
Individual → Collective Learning

One Agent Learns.
Every Agent Benefits.

This answers James's deepest question: how do learnings from one session benefit all agents? The answer is that individual and collective learning are NOT separate systems. They are the same system at different scopes.

The text2sql Example — James's Own Use Case

Step 1 — Raw Query
User asks: "Show me Q4 revenue by region"
Agent generates SQL. Joins fact_revenue + dim_region. Filters on quarter = 'Q4'. Query succeeds. Result returned to user.
Step 2 — Record (Tier 2)
Recorded to KB: question, tables, SQL, outcome
Standard shared memory. Any agent can retrieve this via semantic search. Next similar question gets faster SQL. This is what most systems stop at.
Step 3 — Score (Tier 3: MemRL)
Utility scoring: this pattern scores 0.92
Users kept asking variations of this query. Each successful retrieval increased the utility score. Low-value queries decay. This entry naturally rises to the top of future search results.
Step 4 — Extract (Tier 3: ExpeL)
Rule: "Revenue queries use fact_revenue + dim_region, filter on quarter"
ExpeL extracts a reusable rule with confidence 0.85. This rule fires for ANY revenue-related question, even ones phrased differently. The agent doesn't need to find the original example — it has the pattern.
Step 5 — Prevent (Tier 3: Mistakes)
Tracked: "Don't join on user_id — table uses account_id since migration"
A previous agent made this mistake. It was recorded. Now ANY agent asking a revenue question gets this warning injected BEFORE it starts writing SQL. The mistake happens once. Never again.

Two Scenarios James Described

Scenario 1: Individual + Sub-Agents

One person, AI assistant with sub-agents

Each sub-agent writes findings to KB. The orchestrator learns which sub-agent configurations work best for which tasks (neural router). ExpeL extracts rules from successful task completions. MemRL scores which knowledge actually helped. Individual learning — one person's agent gets smarter over time.

Scenario 2: Team + Shared Agents

Multiple people, shared workflow

All agents share the same KB. Person A's agent discovers "RS256 required for separate deployments" → immediately available to Person B's agent. Mistakes from Monday prevent errors on Tuesday. Collective learning — the whole team gets smarter.

The key insight: Individual and collective learning are NOT separate systems. They're the SAME system at different scopes. The KB is the shared substrate. The learning loop is the shared feedback mechanism. Scale the KB → scale the learning. One person's mistake prevention is every person's mistake prevention.

Insights → Skills → Moat

Deeper Insights Become Skills.

James's insight was right: "deeper insights should be created as a skill." Here's how raw observations compound into an unreplicable advantage through four ascending stages of value.

Stage 1: Observation

"This query pattern works well for revenue questions." Raw KB entry. Unscored. Retrieved by semantic similarity only. Any system can do this.

Stage 2: Pattern

"Revenue queries always need these 3 tables, filtered by quarter." ExpeL rule, confidence 0.85. Fires proactively on matching prompts. Requires learning loop.

Stage 3: Skill

Codified procedure: "When user asks about revenue → use these tables, apply these filters, avoid these joins, prefer these aggregations." Formalized in skill registry. Requires pattern maturity + codification.

The Compounding Effect

Session 1 vs. Session 1,000.

The learning loop doesn't add value linearly. Each mechanism feeds the others. Mistakes prevented create positive compounding — the system avoids entire classes of errors, freeing capacity for novel problems.

Session 1 — Cold Start
Raw prompts, no context, keyword-only routing
KB empty. Preflight returns nothing. Skill routing uses keyword heuristics. Every decision from first principles. Error rate highest.
Sessions 10-50 — Pattern Formation
KB accumulates. First rules extracted. Repeat errors tracked.
MemRL differentiates useful vs. useless knowledge. ExpeL extracts first reusable rules. 17 repeat-error themes populate. Preflight starts returning relevant warnings.
Sessions 50-200 — Router Training
Neural router has enough data. Keyword fallback decreasing.
Hundreds of routing_decisions entries with AES scores. MLX training produces a router outperforming keyword matching. ExpeL rules reach critical mass.
Sessions 200-500 — Institutional Memory
The system knows more than any single session can hold.
Thousands of KB entries with calibrated utility scores. Temporal decay prunes stale knowledge. The system prevents errors the current agent has never seen — because a previous session learned the lesson.
Sessions 500+ — Compound Intelligence
The learning loop IS the competitive moat.
Every new session starts with accumulated intelligence of all prior sessions. Neural router accurately predicts skill requirements. ExpeL rules cover most domains. The system's effective intelligence is the sum of every session that preceded it.
