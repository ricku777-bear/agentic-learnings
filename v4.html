Section 2 — The Missing Layers

The 7 Layers That Make the Difference.

Each layer is shown as a before/after. The before is what your current plan has. The after is what 120 hours of real building revealed was essential. These are not optional enhancements — they are what separates a static reference library from a learning system.

1
The Learning Loop
Feedback writes back to the Knowledge Base
Current Plan

Data flows into the KB. Agent reads from KB. One direction.

Your pipeline is a read-only retrieval system. Data goes in from S3, Kafka, and your sources. The agent queries it. Nothing flows back. Every session starts with the same knowledge it started with on Day 1.

With the Learning Loop

Every session writes back — mistakes, patterns, preferences, insights.

After each session, the system extracts what was learned and writes it to CelerData. Next session, those learnings are injected before work begins. The KB gets smarter with every interaction. This is not manual — it is automatic and mandatory.

Why it matters: "Recording mistakes is not learning. Learning is changed behavior. Without the feedback loop, you have a static reference library, not a learning system." — The mistake files were growing, the same errors kept recurring. The loop only works when writing back is enforced, not optional.
After wiring preflight injection: repeat error rate drops from ~75% to near zero within 30 sessions.
2
Quality Gates — The Forge Pipeline
Worker generates. Observer reviews. Gate validates.
Current Plan

Agent generates output. Output goes to tools. No verification step.

If the agent makes a subtle error in a SQL query, API spec, or configuration — it ships. The only check is human review downstream, which happens after the agent has already committed its work.

With the Forge Pipeline

Three-agent pipeline: Worker produces, Observer reviews with fresh context, Gate validates before delivery.

The Observer has no attachment to the Worker's output — it is adversarial by design. Fresh context window, no shared blind spots. This structure catches errors that single-agent review cannot, because the reviewer does not share the author's assumptions.

Why it matters: "Every enterprise AI fails the same way: appears to work, gets deployed, fails in production invisibly." — A broken launchd plist crashed a production machine. It was caught not by QA but by the sound of the error. Quality gates exist because silent failures are the most expensive kind.
100% correction rate across SQL, Python, Bash, and API spec domains. Errors caught before human review.
3
Trust Engine — Asymmetric Autonomy
Trust earned through demonstrated correctness, not granted by configuration
Current Plan

Permission model: the agent either has access or it doesn't.

Binary permission systems treat a read-only research query the same as a destructive database operation. There is no nuance per action type, no history of correctness factored in, no hard ceiling on irreversible operations.

With the Trust Engine

Asymmetric scoring per category. Trust earned, not configured. Hard ceilings on destructive operations.

+3 per success, -15 per failure. Read operations reach TRUSTED autonomy quickly. DDL, DELETE, financial operations have permanent hard ceilings — no amount of success unlocks them fully. Trust is quantitative, per-category, and evolves through evidence.

Why it matters: "Fully autonomous pipelines fail 41–87% of the time. A single trust level is a single point of failure." — The system that says "done" when nothing was verified is the enterprise AI deployment failure pattern at scale. Trust must be earned through consistent small deliveries, not granted through configuration.
60% of routine work handled autonomously. Human review only for high-risk or novel situations.
4
Context Management — The Relay System
Transfer the WHY, not just the task status
Current Plan

Agent runs until context window fills. Quality degrades silently.

No budget tracking, no handoff protocol, no structured transfer of understanding between sessions. When context fills, work is lost or compacted — and nuance disappears without any visible signal that it has happened.

With the Relay System

Explicit budget: 55% prepare / 70% handoff / 85% emergency. Handoffs transfer the WHY — dead ends, insights, open questions.

Sessions are coordination layers, not workspaces. Heavy work goes to sub-agents with fresh context windows. At handoff thresholds, a structured Problem Brief captures what was learned — not just task status, but the reasoning behind decisions and the dead ends that failed.

Why it matters: "Context degradation is invisible and fatal. HTTP 200 keeps returning while the agent gets progressively dumber." — The relay system was non-functional for 276 consecutive sessions before discovery. Sessions were running 15x over threshold with no visible failure signal. You cannot trust what you cannot measure.
Session context 102KB compressed to 7KB after relay. Zero nuance loss across handoffs.
5
Multi-Agent Orchestration
Parallel by default. Sequential is malpractice at scale.
Current Plan

Single agent, sequential execution. Complex tasks run one step at a time.

A single agent handling research, implementation, review, and documentation in series. The most expensive model handles everything, including tasks that do not require its capabilities.

With Auto-Swarm Orchestration

Tasks decompose automatically into parallel agents. Right model for each job. Cheap models scan, expensive models reason.

Three patterns: research (KB-researcher + web-researcher + codebase-scanner in parallel, then synthesizer), build (researcher + architect feed the main agent, then a critic), analysis (investigator + pessimist + pragmatist in parallel, then synthesizer). Model routing: haiku for search/scan, sonnet for architecture/security, free local models for embeddings and scoring.

Why it matters: "A haiku agent that takes 2 passes is still cheaper than a sonnet agent that does it in 1. Sequential single-agent work on complex tasks is malpractice when you have the capability to parallelize." — Homogeneous multi-agent debate (same model, same prompt) actually performs worse than a single agent. Diversity and adversarial structure are what produce quality — not simply adding more agents.
5–10x throughput improvement on multi-step analysis tasks. Cost discipline enforced at the architecture level.
6
Preflight Intelligence — The Cognitive Layer
Every prompt is analyzed before the agent starts working
Current Plan

Agent receives prompt. Agent starts working. No prior context injected.

Every session starts cold. Past mistakes are not surfaced. Intent is not classified. The agent that made an error yesterday will make the same error today unless someone explicitly reminds it — and no one does.

With the Preflight System

Every prompt runs through intent classification, mistake injection, pattern surfacing, and reasoning mode selection before work begins.

The hook fires before the agent processes the prompt. It queries the KB for relevant past mistakes, classifies intent (directive vs. correction vs. invitation vs. emotional), selects the appropriate reasoning mode, and surfaces relevant skills — all in ~250ms. The agent starts with full context it did not have to ask for.

Why it matters: "Optional self-check is theater. The preflight system forces context into every session, whether the agent wants to see it or not." — Corrections require a different processing path than requests. Emotional weight requires acknowledgment before reasoning. These distinctions change the quality of the output, and they only happen consistently when they are automatic and mandatory.
First-pass quality improvement dramatically measurable within 2 weeks of deployment. Repeat errors approach zero after 30 sessions.
7
Telemetry and Observability
You cannot improve what you cannot measure. You cannot trust what you cannot observe.
Current Plan

No visibility into what agents are doing, how long operations take, or where they fail.

Agent activity is a black box. You know whether a query completed, but not what decisions were made along the way, how long each step took, what was retried, or where confidence was low. Improvements are guesswork.

With Agent Telemetry in StarRocks

Every tool call, artifact creation, and decision emits an event. Real-time dashboards. Circuit breakers for runaway sessions.

A JSONL event stream captures every agent action with structured metadata: tool name, input/output size, latency, session, trust category. This flows directly into StarRocks for real-time analytics. Materialized views surface anomalies. Circuit breakers stop sessions that are consuming 15x expected resources. And critically — you already have the infrastructure for this.

Why it matters: "You can't improve what you can't measure. And you can't trust what you can't observe." — The telemetry advantage at CelerData is structural: you are not building a new analytics layer to observe your agents. You are pointing your existing analytics platform at agent activity. That is a significant implementation shortcut and a competitive moat.
Real-time visibility into all agent activity. Anomalies detected in seconds, not days. Continuous improvement grounded in data, not intuition.

Section 5 — The Sequence

Build Order Matters.

You already have the two hardest parts done. What follows is priority order — not by difficulty but by impact. Each layer unlocks the one after it. The sequence is designed to produce measurable ROI within the first 30 days and compound from there.

Priority Layer When Engineering Effort Why This Order
✓ Done Claude Code as Conductor v1 — complete — Foundation. The conductor model is correct. Do not replace it with a custom framework.
✓ Done Data Pipeline → CelerData KB v1 — complete — Foundation. The KB backbone is correct. StarRocks as vector + full text + semantic layer.
1 Learning Loop (feedback → KB) Week 2–4 Medium Highest leverage. Every session after this one compounds. Without it, every session starts at Day 1.
2 Preflight Intelligence Week 3–5 Medium Prevents repeat errors from the first day the learning loop has data. The loop and the preflight are a pair — one writes, one reads.
3 Quality Gates (Forge) Week 4–6 Medium Catches errors before production. Especially critical for SQL generation, API specs, and configuration artifacts that touch customer systems.
4 Telemetry and Observability Week 4–6 Low (StarRocks!) You already have the infrastructure. Pointing StarRocks at agent event streams is an implementation shortcut, not a new build. Do it early — the data accumulates value over time.
5 Trust Engine Week 6–8 Medium Enables safe autonomous operation. Needs telemetry data to compute meaningful scores — that is why it comes after observability is live.
6 Multi-Agent Orchestration Week 8–12 High Unlocks parallelization and scale. Needs quality gates and trust in place first — autonomous parallel agents without verification is how failures compound.
7 Context Management (Relay) Week 10–14 Medium Prevents degradation at scale. Becomes critical as sessions get longer and tasks get more complex — design it early, implement it when the simpler layers are stable.

The 30-day milestone: Learning Loop + Preflight Intelligence live by Day 30. From that point, every session is better than the last. Repeat error rate begins its measurable decline. The system is now a learning system, not a retrieval system. That is the transition worth celebrating — and everything after it is additive.

The StarRocks shortcut (Layer 4): Telemetry infrastructure that most teams spend 3–6 weeks building from scratch, you already have. StarRocks materialized views, real-time ingestion, and analytical query capabilities map directly onto agent event streams. Treat Layer 4 as a CelerData product advantage — not an engineering task to be sequenced. Ship it as soon as the event schema is defined, not after the other layers.

Support Intelligence

"Every support ticket makes the next resolution faster." Jira, Zendesk, and PagerDuty data flows into the KB and back. Incident patterns surface before the same incident recurs. Resolution time decreases as the system accumulates case history.

Code Review Compounding

"Every code review compounds." Patterns extracted, injected into preflight, repeat errors approach zero. The review system learns what your codebase is prone to and surfaces those patterns before they enter a pull request.

Most routine work — SQL optimization, documentation, incident triage, customer Q&A, code review — handled by agents with appropriate human oversight. Human engineers focus on architecture, novel problems, and judgment calls that require genuine expertise.
