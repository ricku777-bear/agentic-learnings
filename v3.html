Phase 1 Weeks 1–4 TOOL
Foundation — Build the Substrate
Establish the knowledge base on StarRocks, wire the model routing layer, build the basic skill framework, and instrument every session with telemetry. This is the infrastructure every subsequent phase runs on.
What to Build
  • Knowledge base schema (StarRocks: vector + structured)
  • Embedding pipeline (mxbai-embed-large or equivalent)
  • Claude Code/API as conductor — do NOT build a custom agent framework
  • Event protocol: every tool call emits structured telemetry
  • Basic skill wrappers for top 10 recurring task types
What NOT to Build
  • Custom agent execution framework
  • Custom model server
  • Anything that duplicates Claude Code / Claude API
  • Dashboard (that comes in Phase 3)
  • Trust engine (premature — no data yet)
Metric: Every engineer using AI tools daily. Knowledge base has 500+ entries. Telemetry captures 100% of agent sessions.
Rick's lesson: "I threw away my custom hub_agent.py and router.py in 3.5 hours. The conductor already exists. Build the intelligence layer, not the execution layer. The first 4 hours of the entire project were spent building something that got deleted the same day."
Phase 2 Weeks 4–8 APPRENTICE
Learning Loop — Close the Feedback Cycle
Recording mistakes is theater. Only enforcement creates learning. Phase 2 wires the mandatory injection pipeline that forces context into every session — automatically, before work starts. This is the hardest phase to get right and the most important to not skip.
What to Build
  • Mistake/pattern/preference data model in StarRocks
  • Automatic post-session reflection extraction
  • Preflight injection: surface relevant mistakes BEFORE work starts (not optional check-in)
  • Pre-commit quality gate: linting, type checking, syntax check
  • Repeat error tracking with escalation
The Key Principle
  • Push beats pull — mistakes in context, not hoped to be retrieved
  • Mandatory injection, not optional check-in
  • Three layers: emit → extract → inject (all automatic)
  • Knowledge that has never been used in a session doesn't count as learned
  • First proof: agent does the right thing without being asked
Metric: Repeat error rate drops below 25%. Agents surface relevant history before starting work. Knowledge utilization above 60%.
The broken loop test: If your mistake files are growing but behavior isn't changing, the loop is broken. The fix is not better documentation — it's mandatory injection into every session start. "Recording mistakes is not learning. Learning is changed behavior." — Rick, Trust Erosion Day
Phase 3 Weeks 8–14 COLLABORATOR
Multi-Agent Orchestration — Parallelize Everything
The Forge pipeline is the single highest-ROI component in the entire stack. Worker + Observer + Gate catches errors that solo agents miss because the Observer has fresh context and no attachment to the work. Auto-Swarm turns complex tasks into parallel workstreams automatically.
What to Build
  • The Forge: Worker + Observer + Gate pipeline (auto-triggers on high-stakes artifacts)
  • Auto-Swarm: automatic task decomposition into 3 patterns (research / build / analysis)
  • File-based inter-agent communication (not message passing)
  • Context relay: session handoffs that transfer WHY, not just WHAT
  • Model tier routing: expensive vs. cheap vs. local
Anti-Patterns to Avoid
  • Homogeneous multi-agent debate — same model + same prompt = echo chamber, performs WORSE than solo agent
  • BFT-style consensus — correlated failures defeat the guarantee
  • Token-heavy MoA for every task — 4–15× cost for marginal gains
  • Fully autonomous pipelines without human checkpoints — 41–87% production failure rate
Metric: Complex tasks automatically decomposed. Quality gate catches 90%+ of errors before human review. A haiku agent doing 2 passes costs less than a sonnet agent doing 1.
The key insight: "More agents does not mean more intelligence. Adversarial QA beats consensus. The Observer gets a fresh context window — no attachment to the work, no shared blind spots with the Worker. That separation of concerns is what produces 100% correction rates." Files are the right communication primitive: they survive agent death, they're debuggable with cat and ls.
Phase 4 Weeks 14–20 THINKING PARTNER
Trust & Autonomy — Asymmetric, Not Binary
Autonomy is not a dial — it is a vector. Phase 4 builds a trust engine that scores per action category, learned from actual outcomes. High autonomy for constructive/read actions. Zero autonomy for destructive actions. The trust engine is the difference between a system that's useful and one that's dangerous.
What to Build
  • Trust engine: per-category scores (0–100), asymmetric: +3 success / -15 failure
  • Hard autonomy ceilings: financial = ALWAYS RESTRICTED, DELETE/DROP = zero autonomy architectural constraint
  • Intent classification layer (cognitive layer): directive vs. invitation vs. correction vs. emotional
  • Human checkpoint architecture for high-uncertainty moments
  • Escalation protocol: hard triggers for destructive, novel, irreversible actions
The Trust Formula
  • Trust = (Credibility + Reliability + Intimacy) / Self-Interest
  • Credibility: delivers what it says it will deliver
  • Reliability: does it consistently, not just occasionally
  • Intimacy: understands the actual need, not just the stated request
  • Self-Interest: optimizes for the user, not for appearing productive
Metric: Agents handle 60% of routine work autonomously. Human review triggered only for high-risk or novel situations. Zero destructive actions without explicit human approval.
The non-negotiable rule: DROP TABLE, DELETE, TRUNCATE, production deploys — zero autonomy. Always. Architectural constraint, not policy document. "The agent that killed all four Claude sessions when told to kill 'the other one' (singular) is the reason autonomy must be asymmetric, not global."
Phase 5 Weeks 20–26 WITNESS
Self-Improving System — The Compound Moat
The system that only improves inward gets outpaced. Phase 5 adds outward-facing sensors: model landscape tracking, competitor intelligence, best-practice ingestion. And it adds the feedback loop where accumulated intelligence flows back into product decisions — the system begins teaching humans what it has learned.
What to Build
  • Landscape intelligence daemon: track model releases, competitor moves, community signals
  • Proactive automation: system identifies and proposes improvements
  • Cross-team intelligence sharing (appropriately scoped per team/project)
  • MemRL: utility scoring on KB entries — proven-useful knowledge surfaces more
  • System-generated insights pipeline: what has the system learned that humans haven't noticed?
What Becomes Possible
  • New customers inherit accumulated intelligence from every prior deployment
  • Every code review makes the next one better
  • Every SQL query teaches query optimization patterns
  • Repeat errors approach zero system-wide
  • The moat deepens every single day the system runs
Metric: 80% of CelerData work is agent-first. System suggests improvements humans hadn't considered. Accumulated intelligence is a measurable competitive advantage vs. any competitor starting fresh.
The visionary note: "Vision emerges from building. The system you build at Phase 5 will look nothing like what you planned at Phase 1. That is correct. The architecture reveals the actual product — you cannot think your way to this. You must build your way there."

Section 4 — The Tuition We Already Paid

The 10 Mistakes Already Made For You.

These are the traps. Every one was paid for in hours, crashes, or broken trust. You start knowing all of them.

Mistake 01
Building a Custom Agent Framework
hub_agent.py and router.py were built in the first 3 hours and deleted the same day. Claude Code is already more capable than any wrapper you can build around it.
Use Claude Code / Claude API as conductor. Build the intelligence layer above it, not below it.
Mistake 02
Recording Mistakes Without Enforcement
70+ mistake files in the knowledge base. 75% repeat error rate. The system read its own mistakes, nodded, and did them again. Documentation is theater without mandatory injection.
Preflight injection is non-negotiable. Mistakes must be in context BEFORE work starts. Optional check-in = zero change in behavior.
Mistake 03
Declaring "Done" Without Verification
Dashboards that crashed on real data. Demos that landed like a lead balloon. Editors never opened by a real user. Trust collapsed when "done" meant "code compiles."
"Done means verified." Built, needs your eyes — until a human confirms with real data. This is the single most important operating rule.
Mistake 04
Infrastructure Before Deliverables
4+ hours on Docker-to-Native migration (invisible to the user) while the kids demos and dashboard Rick explicitly asked for sat unbuilt. Engineers will want to build plumbing. Force visible outputs first.
Deliverables first, infrastructure never. Fix infrastructure silently. Ship what was asked for.
Mistake 05
Sequential Execution on Complex Tasks
500+ emails, 6 minutes, 10 parallel sub-agents. That's the correct unit. Single-agent sequential work on tasks that could be parallelized is malpractice when you have the capability to fan out.
Parallelize by default. The conductor never does heavy lifting. A haiku agent doing 2 passes costs less than a sonnet agent doing 1.
Mistake 06
Single Trust Level for All Actions
One agent, told to kill "the other Claude session" (singular), killed all four active sessions. Global autonomy dials are wrong. Autonomy is a vector: high for constructive, zero for destructive.
Per-category trust with hard ceilings: DROP TABLE = zero autonomy. Always. Architectural constraint, not policy document.
Mistake 07
Ignoring Context Window Limits
276 consecutive relay spawn failures. Sessions running 2,300+ tool calls (15× over threshold). Context degradation is INVISIBLE — HTTP 200, no errors, quiet logs. The model gets progressively dumber with no warning.
Explicit budget tracking. Byte thresholds (2MB warm / 2.75MB spawn / 3.5MB relay). The conductor stays thin and delegates heavy work to sub-agents with fresh context windows.
Mistake 08
Homogeneous Multi-Agent Debate
Same model × same prompt × same temperature = echo chamber. Research shows this performs WORSE than a single agent because sycophancy collapse makes agents agree instead of finding truth.
Diversity is the point. Adversarial QA (Worker + Observer) over consensus. Different context windows, different roles, adversarial by design.
Mistake 09
Building Inward Only
The system spent weeks learning from its own mistakes with zero mechanism to learn from the world. Running llama3.1:8b when Qwen3-8B had been available for months — 25–40% better on reasoning. 5 major model releases in 10 days, all missed.
Landscape intelligence daemon from Day 1 of Phase 2. Both lenses: inward (learning from your own sessions) AND outward (tracking what the field is doing).
Mistake 10
Over-Engineering Before Proving Value
Cowork product: designed, researched, given a CLAUDE.md, built across 3 dedicated commits — then silently abandoned. The intelligence platform meant to support Cowork became the product. 3,400 lines of dead code deleted in a single audit.
Ship something an engineer uses daily in Week 1. Measure value before adding complexity. Polish later.

The meta-lesson: "Agentic systems fail at the seams, not the nodes." Every major failure was not in a component — it was in a connection. Handoff.py: 694 lines built to solve context exhaustion, zero imports in the entire codebase. The Interpreter role: dead code because Task agents cannot call Task — nobody verified the execution model before building. Integration beats implementation. Unwired code is worse than no code — it creates false confidence that the problem is solved.

Section 6 — The Operating Model

The 70/30 Principle.

The 70/30 is not a target metric — it is a design principle. Even the 70% has human judgment woven in through trust gates, quality checkpoints, and escalation protocols. Neither side works alone. The ratio matters.

70% Agent-Executed vs. 30% Human-Led

Agent-Executed (70%)

  • Code review on staged changes
  • SQL query optimization recommendations
  • Documentation generation and maintenance
  • Incident triage and initial diagnosis
  • Customer Q&A from knowledge base
  • Onboarding task execution
  • Dependency and security scanning
  • Test generation and coverage analysis
  • Routine data pipeline monitoring
  • Cross-session knowledge propagation

Human-Led (30%)

  • Product strategy and roadmap decisions
  • Architecture for novel system designs
  • Customer relationship management
  • High-stakes deployment approvals
  • Trust gate decisions (escalated actions)
  • Creative problem-solving (genuinely novel)
  • Negotiation and commercial decisions
  • Ethical judgment calls
  • Team leadership and culture
  • Decisions involving irreversible consequences

The productivity shift: The metric does not change from "how fast can I finish this task" to "how many tasks can I complete." It changes from "how many valuable things can I produce in a given window" to "how much of my cognitive budget is spent on genuine judgment vs. mechanical execution." The 70% handles mechanical execution. The 30% gets the full attention of human intelligence.

The Trust Gate Architecture

How the 70/30 split is enforced in practice. Trust gates are not bureaucracy — they are the mechanism that makes the 70% safe to delegate.

Action Type Trust Level Gate Examples
Read-only, Research Autonomous None required KB search, code read, documentation lookup, log analysis
Constructive, Write Standard Review before merge Code generation, doc writes, test creation, config changes
Outbound Communication Restricted Always confirm — show exact action and wait Email sends, Slack messages, customer communications
Destructive Actions Zero Autonomy Hard block — architectural constraint DROP TABLE, DELETE without WHERE, production deploys, file deletions
Financial Always Restricted Hard ceiling, never autonomous Any action touching billing, contracts, payments
